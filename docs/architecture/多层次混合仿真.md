# CppTLM 多层次混合仿真平台

## 架构设计文档 — 关键问题与解决方案

**版本 1.0 | 2026-04-03**

---

## 1. 文档概述

本文档针对 CppTLM 多层次混合仿真平台的关键架构挑战，提供具体的设计方案。内容涵盖交易上下文管理、双并行实现策略、多总线协议适配、性能分析可信度保障以及渐进式过渡方法论。

---

## 2. 架构原则

### 2.1 核心愿景

提供统一的 C++ 建模范式，让用户在同一仿真环境中自由组合事务级（TLM）、行为级和周期精确级（RTL）建模。指导原则是：**一次建模，多粒度仿真**。

### 2.2 设计原则

1. **交易穿透**：交易信息（ID 和元数据）必须端到端穿透所有模块接口和微架构层。
2. **接口不变性**：模块级交易接口在不同抽象层级间保持一致，支持无缝切换实现。
3. **协议无关性**：硬件接口支持多拍 Fragment/Stream 协议，适配多种总线标准（AXI、CHI、TileLink）。
4. **双并行实现**：每个模块同时支持 TLM 微模块实现（用于快速性能分析）和 RTL 实现（用于周期精确建模），通过交易 ID 关联。
5. **渐进式过渡**：支持混合运行模式，部分模块使用 TLM 实现、部分使用 RTL 实现，实现增量验证。

## 3. 交易上下文生命周期与作用域管理

### 3.1 问题描述

在多拍 Fragment 场景下，一个交易可能被拆分为多个片段。系统需要追踪父交易与子片段之间的关系，处理中间模块中的交易拆分/合并/重排序，并记录跨交易干扰（如资源竞争）。

### 3.2 解决方案：交易上下文对象

引入 `TransactionContext` 对象，伴随每个交易穿越所有模块边界。这不仅仅是一个 ID，而是一个可携带元数据的数据容器。

#### 3.2.1 TransactionContext 结构

| 字段 | 描述 | 示例 |
|---|---|---|
| `transaction_id` | 交易创建时分配的唯一 ID | `TID_0x0001` |
| `parent_id` | 父交易 ID（用于子交易） | `TID_0x0001` 或空 |
| `fragment_id` | 片段在父交易中的序号 | `0, 1, 2, 3` |
| `fragment_total` | 预期总片段数 | `4` |
| `create_timestamp` | 创建时间（全局虚拟时间） | `0 ps` |
| `source_port` | 来源模块/端口标识 | `cpu_0.out` |
| `type` | 交易类型分类 | `READ/WRITE/ATOMIC` |
| `priority` | QoS 优先级 | `0-7` |
| `trace_log` | 每跳可追加的观测记录 | `[{module, latency}]` |

#### 3.2.2 多拍场景下的 ID 策略

支持两种策略，用户可根据调试粒度需求选择：

1. **同 ID + 片段序号**：所有片段共享父交易的 `transaction_id`，通过不同的 `fragment_id` 区分。适用于高层追踪。
2. **子 ID 生成**：每个片段获得唯一的 `sub_transaction_id`，通过 `parent_id` 关联到父交易。适用于协议级细粒度调试。

#### 3.2.3 交易上下文传播规则

定义三种模块行为来处理交易上下文：

| 模块类型 | 上下文处理规则 |
|---|---|
| **透传型（Passthrough）** | 原样转发 `TransactionContext`，仅向 `trace_log` 追加路由延迟。示例：Crossbar、简单互连。 |
| **转换型（Transformer）** | 创建子交易，分配新的 `sub_transaction_id`，通过 `parent_id` 关联父交易。示例：Cache、地址转换单元。 |
| **终止型（Terminator）** | 标记交易完成，记录最终延迟和状态。示例：PhysicalMemory、终端外设。 |

#### 3.2.4 跨交易干扰记录

当多个交易在模块内部竞争资源时，模块将干扰事件记录到 `TransactionContext.trace_log` 中。每条记录包含：竞争的交易 ID、等待时间、竞争类型（队列满、Bank 忙、仲裁失败）。这使得仿真后可以分析由跨交易交互引起的性能瓶颈。

## 4. 双并行实现策略

### 4.1 问题描述

每个模块接口需要支持两种并行实现：TLM 微模块实现（用于快速性能分析）和 RTL 实现（用于周期精确建模）。两者必须通过交易 ID 关联。挑战包括跨抽象层级的时序对齐、背压行为差异以及状态一致性。

### 4.2 解决方案：实现抽象层

#### 4.2.1 模块接口定义

每个模块定义独立于实现的标准接口契约：

| 接口元素 | 描述 |
|---|---|
| **请求端口 Port\<T\>** | 携带 TransactionContext + 负载的泛型端口。trySend/tryRecv 语义，支持背压。 |
| **响应端口 Port\<T\>** | 携带 TransactionContext + 结果数据的响应端口。支持通过 transaction_id 匹配乱序完成。 |
| **配置接口** | 参数化设置（延迟模型、队列深度、仲裁策略），两种实现共享。 |
| **指标接口** | 统一暴露可观测的微架构指标（命中率、队列深度、平均等待时间）。 |

#### 4.2.2 时间归一化层

TLM 交易可能 1 个 tick 完成，而 RTL 需要 N 个周期。时间归一化层将两个抽象层级的事件映射到统一的逻辑时间线上：

1. **全局虚拟时间（GVT）**：以皮秒为单位的公共时间参考。
2. **TLM 事件**：在交易开始和完成时以 GVT 值打时间戳。
3. **RTL 事件**：在每个时钟沿打时间戳，通过时钟频率映射转换为 GVT。
4. **关联引擎**：按 transaction_id 匹配 TLM 和 RTL 事件，实现延迟、吞吐量和排序的 A/B 对比。

#### 4.2.3 可配置精度开关

为了在 TLM 和 RTL 实现之间进行有意义的 A/B 对比，TLM 模型支持可配置的精度等级：

| 精度等级 | 行为 |
|---|---|
| **快速（默认）** | 每种交易类型固定延迟。不建模背压。最大仿真速度。 |
| **感知背压** | 基于队列的背压，深度可配置。延迟随拥塞变化。 |
| **统计模型** | 延迟从统计分布中抽取，分布参数由 RTL 仿真结果校准。 |

#### 4.2.4 状态一致性验证

如果 TLM 模型维护内部状态（如缓存命中/缺失逻辑），其状态机关键节点必须与 RTL 状态机对应。**状态映射表**定义 TLM 状态与 RTL 状态机节点之间的对应关系。验证框架定期采样两种实现的状态并标记不匹配。

## 5. 多总线 Fragment/Stream 协议适配

### 5.1 问题描述

硬件接口使用多拍 Fragment/Stream 协议，涉及多种总线标准（AXI、CHI、TileLink）。每种总线有独特的握手语义、突发传输规则和乱序支持能力。系统需要统一抽象，能够映射到任意总线协议，同时保持交易 ID 关联。

### 5.2 解决方案：通用协议子集 + 协议适配器

#### 5.2.1 通用协议子集

定义所有总线适配器都实现的最小公共协议：

| 信号 | 方向 | 用途 |
|---|---|---|
| `addr` | Master -> Slave | 交易地址 |
| `data` | 双向 | 数据拍负载 |
| `valid` | Master -> Slave | 数据有效指示 |
| `ready` | Slave -> Master | 背压信号 |
| `last` | Master -> Slave | 最后一拍指示 |
| `transaction_id` | Master -> Slave/响应 | 跨层关联 ID |
| `resp_status` | Slave -> Master | OKAY/EXOKAY/SLVERR/DECERR |

#### 5.2.2 协议适配器架构

每种总线协议（AXI4、CHI、TileLink）实现一个 `ProtocolAdapter`，将其特定信号与通用协议子集互映射：

1. **AXI4 适配器**：将 AW/W/AR/B/R 通道映射到公共信号。`transaction_id` 映射到 AXI `AxID`。乱序响应通过在响应通道匹配 AxID 处理。
2. **CHI 适配器**：将 Req/Rsp/Snp/Dat 通道映射到公共信号。CHI 的 `CompAck` 映射到 ready。`transaction_id` 映射到 CHI `TxnID`。
3. **TileLink 适配器**：将 A/B/C/D/E 通道映射到公共信号。TileLink 的 D 通道天然提供 transaction_id 关联。

#### 5.2.3 乱序完成的 ReorderBuffer

当硬件支持乱序完成时，适配器维护一个以 `transaction_id` 为键的 `ReorderBuffer`：

1. **发送请求时**：在 ReorderBuffer 中分配条目，记录 transaction_id 和请求元数据。
2. **接收响应时**：在 ReorderBuffer 中查找 transaction_id，将响应与原始请求匹配。
3. **乱序处理**：如果响应乱序到达，将其缓存在 ReorderBuffer 中，根据配置按序或乱序释放到上游。
4. **超时检测**：检测停滞交易（在可配置周期数内未收到响应）并标记用于调试。

#### 5.2.4 背压跨协议映射

不同总线协议使用不同的背压信号。适配器将其统一归一化：

| 协议 | 背压映射 |
|---|---|
| **AXI4** | AWREADY/WREADY/ARREADY/BREADY/RREADY 直接映射到公共 ready 信号。当 ready=0 时，上游 trySend 返回 false。 |
| **CHI** | CompAck/LCRdAck/RetAck 结合 Retry 逻辑。Retry 事件映射为临时背压 + 重传请求。 |
| **TileLink** | D 通道 ready 映射到公共 ready。A 通道 NACK 映射为重试，并在 TransactionContext 中设置背压标志。 |

## 6. 性能分析模型可信度

### 6.1 问题描述

早期 TLM 性能分析只有结果可信时才有价值。关键挑战包括：延迟参数来源、流量模型的代表性以及瓶颈识别的粒度。

### 6.2 解决方案：校准的性能建模

#### 6.2.1 延迟模型库

提供可配置的延迟模型库，支持多种参数来源：

1. **历史校准**：来自之前项目的硬件测量数据，存储为参考配置文件。
2. **RTL 派生**：RTL 仿真运行后，自动提取延迟分布并反馈给 TLM 模型进行重新校准。
3. **理论计算**：基于一阶物理估算（如线延迟 = 距离 / 传播速度，缓存命中延迟 = SRAM 访问时间 + 译码器延迟）。
4. **手动覆盖**：用户指定值，用于探索假设配置。

#### 6.2.2 流量生成框架

支持多种流量注入模式，确保分析结果反映真实场景：

| 流量模式 | 描述 |
|---|---|
| **Trace 回放** | 回放真实程序执行 Trace（来自性能计数器或 ISA 仿真器的内存访问序列）。最高保真度。 |
| **合成 Burst** | 可配置的突发模式，定义长度、间隔和地址步长。 |
| **合成 Stride** | 规则访问模式，可配置步长大小，用于缓存行为分析。 |
| **合成 Random** | 均匀或加权随机访问模式，可配置空间/时间局部性。 |
| **混合负载** | 多个不同模式的流量源并发运行，模拟真实的多核/多 Agent 系统。 |

#### 6.2.3 瓶颈识别粒度

TLM 模型可以识别模块级瓶颈，但可能遗漏微架构级根因。解决方案：在 TLM 模块中可选地暴露微架构指标：

- **队列深度监控**：报告所有内部队列的平均值、峰值和填充率。
- **命中/缺失率估算**：对于类缓存结构，使用可配置的统计模型根据访问模式分析估算命中率。
- **平均等待时间**：追踪从请求到达至请求被服务的时间，包含排队延迟。
- **竞争事件**：记录交易被资源竞争阻塞的次数。
- **带宽利用率**：追踪每条链路的实际数据吞吐量与理论最大值之比。

#### 6.2.4 置信度报告

每个性能分析报告为每个指标附带置信度评分：

1. **高置信度**：基于校准的 RTL 数据或历史测量数据。
2. **中置信度**：基于经过验证假设的理论计算。
3. **低置信度**：基于默认估算或手动覆盖，需要 RTL 验证。

## 7. 渐进式过渡策略

### 7.1 问题描述

从 TLM 性能分析到 RTL 周期精确建模的过渡应该是渐进式的，而非一次性替换。系统必须支持混合运行模式，并在每个步骤提供回归验证。

### 7.2 解决方案：混合运行与契约测试

#### 7.2.1 模块级实现选择

每个模块通过配置独立选择实现类型：

| 配置 | 行为 |
|---|---|
| `impl_type = "tlm"` | 使用 TLM 微模块实现。快速仿真，近似时序。 |
| `impl_type = "rtl"` | 使用 RTL（CppHDL）实现。周期精确，仿真速度较慢。 |
| `impl_type = "compare"` | 并行运行两种实现。对比输出验证功能等价性。开发和验证模式。 |
| `impl_type = "shadow"` | RTL 以影子模式运行（接收输入但输出丢弃）。记录 RTL 行为用于对比，不影响系统行为。 |

#### 7.2.2 接口契约测试

为确保 TLM 和 RTL 实现在功能上等价（即使时序不同），定义契约测试框架：

1. **输入序列**：向两种实现发送相同的交易序列。
2. **功能输出**：验证两者对每个 transaction_id 产生相同的响应数据。
3. **排序**：如果需要有序响应，验证排序一致。
4. **背压**：验证两者正确处理背压（TLM 使用感知背压精度模式）。
5. **边界条件**：测试边界情况（队列满、无效地址、协议违规）。

#### 7.2.3 回归框架

每个 RTL 实现完成后，针对 TLM 基线运行回归测试套件：

1. **功能回归**：新 RTL 模块通过所有契约测试。
2. **性能回归**：对比 RTL 延迟/吞吐量与 TLM 基线。标记超出可配置阈值（如 20%）的偏差。
3. **混合系统回归**：将新 RTL 模块与其他 TLM 模块一起运行全系统仿真。验证系统级正确性。
4. **内存安全**：使用 AddressSanitizer 运行，检测长时间运行仿真中的内存泄漏或缓冲区溢出。

#### 7.2.4 推荐过渡序列

1. **阶段一：全 TLM** — 使用 TLM 实现完成系统级架构探索。
2. **阶段二：关键路径 RTL** — 将最关键的性能模块（如缓存控制器、内存控制器）替换为 RTL 实现。
3. **阶段三：扩展 RTL 覆盖** — 随着微架构设计成熟，逐步将更多模块替换为 RTL。
4. **阶段四：全 RTL** — 所有模块使用 RTL 实现进行最终的流片验证。

## 8. 完整架构概览

### 8.1 分层架构

| 层级 | 组件与职责 |
|---|---|
| **应用层** | 流量生成器、Trace 回放、负载注入、性能仪表盘、置信度报告。 |
| **模块接口层** | 携带 TransactionContext 的 Port\<T\>。请求/响应端口。指标接口。配置接口。 |
| **实现层** | TLM 微模块（快速近似）。RTL 微模块（通过 CppHDL 周期精确）。实现选择器/路由器。 |
| **适配层** | TLMToStreamAdapter、TLMToFlowAdapter、ProtocolAdapter（AXI/CHI/TileLink）、ReorderBuffer、时间归一化器、零拷贝内存代理。 |
| **仿真内核层** | 多时钟调度器、全局虚拟时间、统一事件队列、物理内存池。 |
| **CppHDL 集成层** | HybridComponentWrapper、ch::Stream/ch::Flow 集成、CppHDL 仿真器桥接、标准组件库。 |

### 8.2 数据流：端到端交易示例

从 CPU 经 Cache 到 Memory 的完整交易流程：

1. **CPU（TLM）** 创建 `ReadRequest`，携带 `TransactionContext`（TID=0x0001，source=cpu_0.out，create_timestamp=0）。
2. **CPU** 通过 `Port<ReadRequest>` 发送到 `TLMToStreamAdapter`。
3. **适配器** 将 TransactionContext + 负载转换为 CppHDL Bundle 信号，通过 ProtocolAdapter 映射到 AXI4。`transaction_id` 映射到 AxID。
4. **AXI4 StreamArbiter** 在多个 Master 间仲裁。背压传播到适配器 FIFO。
5. **Cache RTL（CppHDL）** 接收 AXI4 请求。命中：1 个周期处理完成，生成相同 TID 的响应。未命中：创建子交易（parent_id=TID_0x0001，sub_id=TID_0x0002）发往 L2。
6. **L2 总线（TLM）** 通过 `RTLToTLMApdapter` 接收子交易。转发到 PhysicalMemory。
7. **PhysicalMemory（TLM）** 终止交易。响应沿链路返回，每个模块向 trace_log 追加延迟。
8. **CPU** 收到 `ReadResponse`，TID=0x0001。完整 trace_log 显示：`cpu(0) -> adapter(2) -> cache_hit(1)` 或 `cache_miss->l2(15)->mem(50) -> 返回路径`。

## 9. 开发计划

### 9.1 阶段一：交易上下文与基础接口（第 1-4 周）

1. 实现 `TransactionContext` 结构体，包含第 3.2.1 节定义的所有字段。
2. 实现 `Port<T>` 模板，支持 trySend/tryRecv 语义和 TransactionContext 传播。
3. 实现透传/转换/终止传播规则作为模块基础行为。
4. 开发 trace_log 中的跨交易干扰记录功能。

**里程碑**：两个 TLM 模块通过 `Port<T>` 通信，实现完整的 TransactionContext 传播和追踪日志。

### 9.2 阶段二：适配层核心（第 5-8 周）

1. 实现 `TLMToStreamAdapter`，包含 FIFO 缓冲和 valid/ready 握手。
2. 实现 `TLMToFlowAdapter`，用于简单的事件驱动通信。
3. 实现通用协议子集信号和 `ProtocolAdapter` 基类。
4. 实现 AXI4 ProtocolAdapter 作为首个具体协议实现。
5. 实现 `ReorderBuffer` 用于乱序响应匹配。

**里程碑**：TLM 模块 -> TLMToStreamAdapter -> AXI4 ProtocolAdapter -> CppHDL Stream 数据流验证通过，transaction_id 关联正确。

### 9.3 阶段三：双并行实现与时间归一化（第 9-12 周）

1. 实现实现抽象层，支持 `impl_type` 配置。
2. 实现时间归一化层，包含全局虚拟时间映射。
3. 实现可配置精度开关（快速/感知背压/统计模型）。
4. 实现状态映射表和一致性验证引擎。
5. 实现 `compare` 和 `shadow` 模式，支持 TLM/RTL 并行执行。

**里程碑**：单模块以 compare 模式运行 — TLM 和 RTL 产生关联输出，按 transaction_id 匹配，生成时间归一化的对比报告。

### 9.4 阶段四：协议库与复杂拓扑（第 13-16 周）

1. 实现 CHI ProtocolAdapter，包含 CompAck/Retry 映射。
2. 实现 TileLink ProtocolAdapter。
3. 实现跨协议桥接（如 AXI-to-CHI），支持异构互连。

## 10. 风险登记与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| **交易 ID 耗尽** | 长时间仿真中，交易 ID 空间可能溢出。 | 使用 64 位交易 ID。实现已完成交易的 ID 回收。添加 ID 池监控。 |
| **适配器 FIFO 溢出** | 高负载下，适配器 FIFO 可能溢出导致数据丢失。 | 可配置 FIFO 深度与运行时监控。自动背压传播。溢出检测与错误报告。 |
| **跨时钟域缺陷** | GemSc 与 CppHDL 时钟域之间的 CDC 可能引入微妙的时序缺陷。 | 适配器逻辑的形式化 CDC 验证。异步 FIFO 模型与亚稳态仿真。CDC 专项测试用例。 |
| **性能下降** | 混合 TLM/RTL 仿真速度可能过慢，无法实用。 | 非关键路径使用 TLM Bypass 模式。可配置的 RTL 仿真粒度。多线程适配器处理。 |
| **协议一致性差距** | 协议适配器可能不完全符合总线规范。 | 每个适配器的协议一致性测试套件。参考模型对比。增量特性覆盖率追踪。 |
| **调试复杂度** | 混合 TLM/RTL 调试本身就很复杂。 | 带 transaction_id 注释的统一 VCD。分层调试视图（TLM 视图、RTL 视图、关联视图）。自动化异常检测。 |
