## 一、产品需求核心内容（已整理）

### 1. 产品愿景

**一次建模，多粒度仿真** — 在同一 C++ 框架下，支持事务级 (TLM) 和周期精确级 (RTL) 建模的无缝混合仿真。

### 2. 目标用户（需确认）

| 用户类型 | 需求场景 | 优先级 |
|---------|---------|--------|
| 芯片架构师 | 早期架构探索，快速评估不同设计选项 | P0 |
| RTL 设计师 | 在系统级上下文中验证 RTL 模块 | P1 |
| 软件开发人员 | 在 RTL 完成前开始软件开发 | P1 |
| 验证工程师 | 混合仿真环境下的功能验证 | P2 |

### 3. 核心功能需求（需确认优先级）

| ID | 需求描述 | 来源 | 优先级 |
|----|---------|------|--------|
| **FR-001** | TLM 模块和 RTL 模块在同一仿真中混合运行 | 多层次混合仿真.md §2 | P0 |
| **FR-002** | 交易上下文端到端追踪（transaction_id 穿透） | 多层次混合仿真.md §3 | P0 |
| **FR-003** | 支持模块级实现选择（tlm/rtl/compare/shadow） | 多层次混合仿真.md §4 | P1 |
| **FR-004** | 支持 AXI4 协议适配器 | 多层次混合仿真.md §5 | P1 |
| **FR-005** | 支持 CHI 协议适配器 | 多层次混合仿真.md §5 | P2 |
| **FR-006** | 支持 TileLink 协议适配器 | 多层次混合仿真.md §5 | P2 |
| **FR-007** | 性能分析报告，带置信度评分 | 多层次混合仿真.md §6 | P2 |
| **FR-008** | VCD 波形导出，带 transaction_id 注释 | 多层次混合仿真.md §7 | P2 |

### 4. 非功能性需求（需确认目标值）

| ID | 需求描述 | 当前提案值 | 需确认 |
|----|---------|-----------|--------|
| **NFR-001** | 向后兼容 | 现有代码零修改 | ✅ |
| **NFR-002** | 性能开销 | < 10% | ❓ |
| **NFR-003** | 仿真速度 | 纯 TLM > 1000 KIPS | ❓ |
| **NFR-004** | 延迟预测精度 | < 15% 误差 | ❓ |
| **NFR-005** | 单元测试覆盖率 | > 90% | ✅ |

### 5. 实施路线图（需确认）

| Phase | 内容 | 提案周期 | 需确认 |
|-------|------|---------|--------|
| **Phase 0** | 环境准备 | 1 周 | ❓ |
| **Phase 1** | TransactionContext + Port\<T\> | 3 周 | ❓ |
| **Phase 2** | StreamBridge + FragmentMapper | 4 周 | ❓ |
| **Phase 3** | 双并行实现 | 4 周 | ❓ |
| **Phase 4** | 协议库扩展 (CHI/TileLink) | 4 周 | ❓ |
| **Phase 5** | 产品化 | 4 周 | ❓ |

## 二、待讨论议题清单

### 议题 1：目标用户优先级

**背景**: 文档中提到 4 类用户，但资源有限，需要明确优先级。

**问题**:
1. 哪类用户是首要服务对象？
2. 是否有其他目标用户未列出？

**选项**:
- A) 芯片架构师优先（早期架构探索）
- B) RTL 设计师优先（系统级验证）
- C) 软件开发者优先（软硬件并行开发）
- D) 平均支持，无明显优先级

---

### 议题 2：核心功能范围

**背景**: 提案中包含 8 个功能需求，但实施资源有限。

**问题**:
1. 哪些功能是 MVP（最小可行产品）必须的？
2. 哪些功能可以延后到 v2.0？

**选项**:
- A) MVP 仅包含 FR-001 + FR-002（混合运行 + 交易追踪）
- B) MVP 包含 FR-001~004（增加模块实现选择和 AXI4）
- C) 全部功能都是 P0，无延后

---

### 议题 3：协议适配器优先级

**背景**: 提案支持 3 种总线协议（AXI4, CHI, TileLink）。

**问题**:
1. 首期实现哪个协议？
2. CHI 和 TileLink 的优先级如何？

**选项**:
- A) 仅 AXI4（覆盖主流场景）
- B) AXI4 + TileLink（开源生态）
- C) AXI4 + CHI（服务器/数据中心场景）
- D) 三者并行（资源充足时）

---

### 议题 4：性能指标目标值

**背景**: 提案中给出了性能指标目标值。

**问题**:
1. 性能开销 < 10% 是否合理？
2. 仿真速度 > 1000 KIPS 是否必要？
3. 延迟预测精度 < 15% 误差是否可接受？

**选项**:
- A) 维持提案值
- B) 放宽要求（开销 < 20%，速度 > 500 KIPS，精度 < 25%）
- C) 更严格要求（开销 < 5%，速度 > 2000 KIPS，精度 < 10%）

---

### 议题 5：实施时间表

**背景**: 提案总周期 20 周（约 5 个月）。

**问题**:
1. 这个时间表是否符合你的预期？
2. 是否有明确的交付截止日期？
3. 是否需要压缩或延长某些 Phase？

**选项**:
- A) 维持 20 周计划
- B) 压缩到 12-16 周（减少功能范围）
- C) 延长到 24-30 周（保证质量）
- D) 分两期：第一期 10 周（核心功能），第二期 10 周（扩展功能）

---

### 议题 6：成功指标

**背景**: 如何衡量产品成功？

**问题**:
1. 除了技术指标，还有哪些业务指标？
2. 是否有具体的应用场景或标杆项目？

**选项**:
- A) 技术指标为主（性能、覆盖率等）
- B) 用户满意度为主（内部调研 > 4/5）
- C) 项目落地为主（至少 1 个实际项目使用）
- D) 综合指标（技术 + 用户 + 项目）

---

### 议题 7：风险接受度

**背景**: 已识别 6 个主要风险。

**问题**:
1. 哪些风险是必须缓解的？
2. 哪些风险可以接受？

**风险列表**:
| 风险 | 提案缓解措施 |
|------|-------------|
| TransactionContext 性能开销 > 5% | 基准测试验证，优化内存布局 |
| CppHDL 接口变更 | 锁定 commit hash，编写接口测试 |
| 命名空间冲突 | 明确命名空间封装，代码审查 |
| 时间归一化复杂度高 | Phase 1 简化方案，Phase 3 扩展 |
| 分片重组逻辑缺陷 | 契约测试 + 边界条件测试 |
| 仿真速度过慢 (<100 KIPS) | 非关键路径 TLM Bypass 模式 |
